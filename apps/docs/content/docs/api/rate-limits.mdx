# Rate Limits

The Sim API implements rate limiting to ensure fair usage and optimal performance for all users. Rate limits vary based on your authentication method and subscription plan.

## Rate Limit Headers

Every API response includes rate limit information in the headers:

```http
HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1640995200
X-RateLimit-Window: 60
```

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |
| `X-RateLimit-Window` | Window duration in seconds |

## Rate Limit Exceeded Response

When you exceed the rate limit, you'll receive a `429 Too Many Requests` response:

```json
{
  "error": "Rate limit exceeded",
  "message": "API rate limit exceeded. Try again in 45 seconds.",
  "requestId": "req_abc123",
  "retry_after": 45,
  "limit": 1000,
  "window": 60
}
```

## Best Practices

### 1. Monitor Rate Limit Headers

Always check rate limit headers and implement logic to handle limits gracefully:

```javascript
async function makeApiRequest(url, options = {}) {
  const response = await fetch(url, options);
  
  // Check rate limit headers
  const limit = response.headers.get('X-RateLimit-Limit');
  const remaining = response.headers.get('X-RateLimit-Remaining');
  const reset = response.headers.get('X-RateLimit-Reset');
  
  console.log(`Rate limit: ${remaining}/${limit}, resets at ${new Date(reset * 1000)}`);
  
  if (response.status === 429) {
    const retryAfter = response.headers.get('Retry-After');
    console.log(`Rate limited. Retry after ${retryAfter} seconds`);
    throw new Error(`Rate limited. Retry after ${retryAfter} seconds`);
  }
  
  return response;
}
```

### 2. Implement Exponential Backoff

When you hit rate limits, use exponential backoff for retries:

```javascript
async function apiRequestWithBackoff(url, options = {}, maxRetries = 3) {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url, options);
      
      if (response.status === 429) {
        if (attempt === maxRetries) {
          throw new Error('Max retries exceeded');
        }
        
        const retryAfter = parseInt(response.headers.get('Retry-After') || '1');
        const delay = Math.min(retryAfter * 1000, Math.pow(2, attempt) * 1000);
        
        console.log(`Rate limited. Waiting ${delay}ms before retry ${attempt + 1}`);
        await new Promise(resolve => setTimeout(resolve, delay));
        continue;
      }
      
      return response;
    } catch (error) {
      if (attempt === maxRetries) throw error;
      
      const delay = Math.pow(2, attempt) * 1000;
      console.log(`Request failed. Waiting ${delay}ms before retry ${attempt + 1}`);
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
}
```

### 3. Batch Operations

Use bulk endpoints when available to reduce the number of API calls:

```javascript
// Instead of multiple individual requests
for (const workflowId of workflowIds) {
  await deleteWorkflow(workflowId);
}

// Use bulk operations
await bulkDeleteWorkflows(workflowIds);
```

### 4. Cache Frequently Used Data

Implement caching to reduce API calls:

```javascript
class WorkflowCache {
  constructor(ttl = 300000) { // 5 minutes default
    this.cache = new Map();
    this.ttl = ttl;
  }
  
  async getWorkflow(id) {
    const cached = this.cache.get(id);
    
    if (cached && Date.now() - cached.timestamp < this.ttl) {
      return cached.data;
    }
    
    const workflow = await fetchWorkflowFromApi(id);
    this.cache.set(id, {
      data: workflow,
      timestamp: Date.now()
    });
    
    return workflow;
  }
}
```

### 5. Use Webhooks Instead of Polling

For real-time updates, use webhooks instead of frequently polling the API:

```javascript
// Bad: Frequent polling
setInterval(async () => {
  const status = await checkWorkflowStatus(workflowId);
  if (status === 'completed') {
    handleCompletion();
  }
}, 5000); // This will quickly hit rate limits

// Good: Use webhooks
app.post('/webhook/workflow-completion', (req, res) => {
  const { workflowId, status } = req.body;
  if (status === 'completed') {
    handleCompletion(workflowId);
  }
  res.status(200).send('OK');
});
```

### Programmatic Monitoring

Track your usage programmatically:

```javascript
class RateLimitMonitor {
  constructor() {
    this.usage = {
      requests: 0,
      limit: 0,
      remaining: 0,
      resetTime: 0
    };
  }
  
  updateFromHeaders(headers) {
    this.usage = {
      requests: this.usage.requests + 1,
      limit: parseInt(headers.get('X-RateLimit-Limit') || '0'),
      remaining: parseInt(headers.get('X-RateLimit-Remaining') || '0'),
      resetTime: parseInt(headers.get('X-RateLimit-Reset') || '0')
    };
  }
  
  getUsagePercent() {
    return ((this.usage.limit - this.usage.remaining) / this.usage.limit) * 100;
  }
  
  shouldWarn() {
    return this.getUsagePercent() > 80; // Warn at 80% usage
  }
  
  timeUntilReset() {
    return Math.max(0, this.usage.resetTime - Math.floor(Date.now() / 1000));
  }
}
```

## Handling Different Scenarios

### High-Volume Data Processing

For processing large amounts of data:

```javascript
class BatchProcessor {
  constructor(batchSize = 10, delayMs = 1000) {
    this.batchSize = batchSize;
    this.delayMs = delayMs;
  }
  
  async processBatch(items, processor) {
    const results = [];
    
    for (let i = 0; i < items.length; i += this.batchSize) {
      const batch = items.slice(i, i + this.batchSize);
      
      const batchResults = await Promise.all(
        batch.map(item => processor(item))
      );
      
      results.push(...batchResults);
      
      // Delay between batches to stay within rate limits
      if (i + this.batchSize < items.length) {
        await new Promise(resolve => setTimeout(resolve, this.delayMs));
      }
    }
    
    return results;
  }
}
```

### Real-time Applications

For real-time applications that need frequent updates:

```javascript
class RealTimeUpdater {
  constructor(apiClient) {
    this.apiClient = apiClient;
    this.updateQueue = [];
    this.isProcessing = false;
  }
  
  queueUpdate(workflowId, changes) {
    this.updateQueue.push({ workflowId, changes, timestamp: Date.now() });
    this.processQueue();
  }
  
  async processQueue() {
    if (this.isProcessing) return;
    this.isProcessing = true;
    
    while (this.updateQueue.length > 0) {
      // Check rate limit before processing
      const usage = await this.apiClient.getRateLimitStatus();
      if (usage.remaining < 10) {
        // Wait for rate limit reset if we're running low
        await new Promise(resolve => 
          setTimeout(resolve, (usage.resetTime - Date.now()) + 1000)
        );
      }
      
      const update = this.updateQueue.shift();
      try {
        await this.apiClient.updateWorkflow(update.workflowId, update.changes);
      } catch (error) {
        if (error.status === 429) {
          // Put the update back in the queue
          this.updateQueue.unshift(update);
          await new Promise(resolve => setTimeout(resolve, 5000));
        }
      }
    }
    
    this.isProcessing = false;
  }
}
```
